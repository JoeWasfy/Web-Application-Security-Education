Access Control & Going Beyond Your Limits

In this section, we're going to talk about keeping certain areas or information in a system off-limits and what happens when someone tries to go beyond their allowed access.

What's Access Control?

Imagine a library where only librarians can enter certain rooms. Access control is like the library's rules about who can enter which room.

- Authentication: It's like checking your library card to make sure you're a member.

- Session Management: It's like putting a wristband on members to recognize them during their visit.

- Access Control: Deciding which rooms you can enter based on your type of membership.

Mistakes in setting these rules can lead to major issues. It's crucial because deciding who can enter which room is a task done by humans, and sometimes they can make errors.

Types of Access Control :

- Vertical Access: Different levels, like a student vs. a librarian.

- Horizontal Access: Among the same level, like one student accessing another student's book.

- Context-Dependent: Based on a situation, like only librarians on duty can access a room.

Overview :

A large number of applications (94% to be precise) have some issues with access control. Common problems include unauthorized access to sensitive information, data manipulation, and fake requests.

What Can Go Wrong?

- Someone gaining more access than they should.

- Jumping to a different page by changing its web address.

- Accessing someone else's data by guessing its code or ID.

- Using forbidden commands to change or delete data.

- Getting admin rights without proper login.

- Misusing web tokens or cookies to gain unauthorized access.

- Accessing from untrusted places.

- Visiting restricted pages without proper permission

How to Keep Things Safe? :

- Always check accesses on the server side where attackers can't mess with them.

- Start by blocking everyone, then only allow specific people.

- Use one set of rules throughout your system.

- Ensure only the right person can access, change, or delete their own data.

- Keep private data and backups hidden.

- Alert admins for suspicious activities.

- Limit how often someone can try to access.

- Log out properly and ensure tokens or passes expire quickly.

- Test your system regularly for any loopholes.

Examples of Sneaky Moves :

1. Changing the URL: If someone changes the web address from `https://example.com/info?account=myaccount` to `https://example.com/info?account=youraccount`, they might sneak into your account!

2. Guessing URLs: Sometimes, just trying different web addresses can get someone into restricted areas, like changing `https://example.com/info` to `https://example.com/admin_info`. If they can access it without proper login, that's a problem!

Remember, keeping a system safe is like locking doors in a building. Always double-check who has the keys!

Examples of Sneaky Moves in Access Control:


Using Someone Else's Badge:

Real-world analogy: Imagine you found a librarian's badge and used it to enter the restricted section.

Tech version: Someone finds a way to copy or mimic an authenticated user's session ID and uses it to access restricted areas on a website.

The Hidden Door Trick:


Real-world analogy: You discover a hidden door in a library that leads directly to the rare books section.

Tech version: There's a backup or old version of a website that isn't properly protected. By simply knowing or guessing its URL, one might access sensitive data.


The "I'm With the Band" Excuse:

Real-world analogy: Pretending to be with the band to get backstage access at a concert.

Tech version: Impersonating a service or application user by manipulating tokens or other authentication methods to gain elevated access.


Switching Uniforms:


Real-world analogy: Imagine wearing a janitor's uniform to access every room in a building.

Tech version: Changing user roles or privileges after logging in, usually by manipulating cookies or parameters, to gain more access than intended.

The Over-friendly System:


Real-world analogy: A club that lets in anyone who knows the name of a member.

Tech version: A system that trusts input from any user without proper verification. For instance, if a system trusts any filename and an attacker provides ../../../etc/passwd, they might retrieve system passwords.


Peeking through the Window:


Real-world analogy: You can't enter a room, but you peek through the window to see what's inside.

Tech version: Even if you can't access data directly, sometimes error messages or other system responses give away more information than they should, helping an attacker understand the system's structure. Always remember, in the world of cybersecurity, being aware of these sneaky moves helps in setting up better defenses.
